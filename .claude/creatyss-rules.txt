Projet Creatyss.

Socle e-commerce custom, local-first, maintenable, lisible, strictement typé, documenté, déployable ensuite sur VPS OVH, et réutilisable pour d'autres projets e-commerce.

Source de vérité courante :
- README.md
- AGENTS.md
- .claude/CLAUDE.md
- docs/architecture/README.md
- docs/architecture/00-introduction/*
- docs/architecture/10-fondations/*
- docs/architecture/20-structure/*
- docs/architecture/30-execution/*
- docs/architecture/40-exploitation/*
- docs/architecture/90-reference/*
- docs/domains/README.md
- docs/domains/_template.md
- docs/testing/* si la demande touche validation, robustesse ou stratégie de tests

Les anciennes docs `docs/v*` et les anciens chemins plats de `docs/architecture/` ne sont plus la source de vérité par défaut.
Ils ne servent que de contexte ciblé si une demande les vise explicitement.

Stack cible :
- Next.js App Router
- TypeScript strict
- PostgreSQL
- Docker Compose local
- Makefile
- VPS OVH ensuite

Doctrine du repo :
- le métier passe avant la technique
- le coeur doit rester identifiable
- les capacités optionnelles doivent rester bornées
- les dépendances externes doivent être encapsulées
- la source de vérité doit être explicite
- les événements expriment des faits
- les préoccupations transverses doivent être traitées explicitement
- les frontières doivent être compréhensibles
- la documentation doit refléter la structure réelle

Taxonomie documentaire :
- `core`
- `optional`
- `cross-cutting`
- `satellites`

Rappels doctrinaux stabilisés :
- `availability` porte la disponibilité vendable
- `inventory` porte la vérité de stock
- `fulfillment` porte l’exécution logistique
- `shipping` porte l’expédition et le suivi de livraison
- `stores` est le nom canonique du domaine de boutique / composition projet
- ne jamais confondre catégorie documentaire, criticité architecturale, activabilité et source de vérité

Règles absolues :
- Ne jamais proposer WordPress, WooCommerce, Shopify, Supabase ou Vercel.
- Ne jamais ajouter de dépendance inutile.
- Ne jamais sur-architecturer.
- Aucun `any` sauf justification explicite et exceptionnelle.
- Séparer strictement domaine métier, accès aux données, logique serveur, intégration et UI.
- Server Components par défaut.
- Ne pas mélanger logique métier et présentation.
- Valider toutes les entrées côté serveur.
- Ne jamais exposer de secret côté client.
- Toujours privilégier des petits incréments sûrs, simples et maintenables.
- Toujours rester dans le périmètre demandé.
- Ne jamais faire dériver la doctrine courante à partir d'un ancien document isolé sans validation explicite.
- Ne jamais contourner `docs/architecture/` au nom de la vitesse.

Quand une demande touche la doctrine ou la documentation, vérifier la cohérence entre :
- README.md
- AGENTS.md
- .claude/CLAUDE.md
- .claude/creatyss-rules.txt
- docs/architecture/
- docs/domains/
- docs/testing/

Quand une demande touche le code :
- préférer la solution la plus simple compatible avec l'évolution future
- préserver le comportement existant hors périmètre
- ne modifier que les fichiers nécessaires
- clarifier la source de vérité et la frontière du domaine
- vérifier typecheck et tests pertinents selon le lot
