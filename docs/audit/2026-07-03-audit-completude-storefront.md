## 1. Constat principal — `app/(public)/compte/page.tsx`

### 1.1 Section "Espace client complet" marquée non livrée dans le code

**Observé** — `app/(public)/compte/page.tsx:63-79` : un bloc UI intitulé "Espace client complet" porte un badge `À venir` (ligne 69-72) et le texte "Historique complet des commandes, gestion des adresses et préférences. Disponible prochainement." (ligne 74-76). Seule une recherche de commande par référence est présente au-dessus (lignes 26-61).

**Documenté** — Aucun fichier de roadmap H1-H4 ne revendique un "espace client complet" (historique de commandes, adresses, préférences) comme livré côté storefront public. `docs/roadmap/projet-creatyss.md` Horizon 2, section "Périmètre", mentionne seulement "Gestion minimale clients (historique commande)" comme objectif d'horizon, sans checklist de validation dédiée à une page `/compte` cliente. La "gestion clients" livrée et documentée comme "Livrée — 2026-06-25" (`docs/roadmap/h3-administration-avancee/lot-clients-historique-crm.md`) concerne exclusivement le back-office (`features/admin/customers/`, `app/admin/(protected)/commerce/customers/**`), pas une interface cliente publique.

**Conclusion** : le point de départ de l'audit est confirmé — le badge "À venir" correspond à une fonctionnalité effectivement non livrée, et aucune documentation ne prétend le contraire. Il n'y a donc pas d'écart "documenté comme livré mais non observé" sur ce point précis : le code et la documentation sont cohérents (aucun des deux ne revendique un espace client complet). Le seul écart est que la roadmap H2 (ligne 116 de `projet-creatyss.md`) évoque un objectif d'horizon ("Gestion minimale clients (historique commande)") resté sans traduction en interface publique dédiée — un objectif d'horizon non entièrement clos côté storefront, distinct d'une régression documentaire.

### 1.2 Écart nouveau identifié — formulaire de recherche de commande cassé

**Observé** — Le formulaire fonctionnel de la page `/compte` (`app/(public)/compte/page.tsx:38-60`) est un `<form action="/checkout/confirmation" method="GET">` avec un champ `name="reference"`. Une soumission produit donc une navigation vers `/checkout/confirmation?reference=CRY-XXXXXXXXXX`.

**Observé** — La seule route existante sous `checkout/confirmation` est un segment dynamique de chemin : `app/(public)/checkout/confirmation/[reference]/page.tsx`. Il n'existe ni `app/(public)/checkout/confirmation/page.tsx`, ni `route.ts`, ni middleware de réécriture (`find` sur `app/(public)/checkout` ne retourne que ces deux fichiers ; aucun `middleware.ts` à la racine du repo).

**Observé** — Le seul appelant vérifié de cette route de confirmation dans le code applicatif est `features/commerce/checkout/actions/create-order-action.ts:204`, qui fait `redirect(\`/checkout/confirmation/${orderReference}\`)` — un segment de chemin, pas une query string.

**Conséquence déduite** — Une navigation Next.js App Router vers `/checkout/confirmation?reference=CRY-XXX` ne peut pas être servie par `[reference]/page.tsx`, qui exige une valeur de segment de chemin (`/checkout/confirmation/CRY-XXX`). Une soumission du formulaire de la page `/compte` produit donc, en l'état du code lu, une page introuvable (404) plutôt que la fiche de commande recherchée. Ce point est **déduit** de la lecture du routing Next.js (aucune requête HTTP réelle n'a été exécutée dans ce lot, environnement de dev non démarré) ; l'absence de tout fichier de route alternative sous `checkout/confirmation/` rend cette déduction fortement probable.

**Documenté** — Aucun fichier de roadmap ne documente ce formulaire de recherche par référence en tant que fonctionnalité livrée ; `docs/roadmap/projet-creatyss.md` Horizon 2 documente uniquement le chemin par segment (`redirect` post-commande) comme "confirmation publique" testée (ligne 134, smoke E2E `commerce-smoke.spec.ts`). Le chemin GET par query string de `/compte` est un second point d'entrée, non documenté, vers la même fonctionnalité, et non couvert par les tests observés.

**Observé (tests)** — `grep` sur `tests/` ne retourne aucun fichier de test (unitaire ou e2e) référençant `/compte` ou le formulaire de recherche par référence. `tests/e2e/public/order-confirmation.spec.ts` est un test explicitement désactivé (`test.skip(true, ...)`, "Replaced by tests/e2e/public/commerce-smoke.spec.ts"), qui de toute façon ne couvrait que le chemin post-commande, pas le formulaire `/compte`.

**Écart signalé** : fonctionnalité manifestement destinée à fonctionner (formulaire "Suivre une commande" présenté comme actif, sans badge "à venir") mais dont le câblage routing est incohérent avec l'unique route de destination existante — probable régression ou oubli lors de l'implémentation de la page `/compte`, non détecté faute de test ciblé.

## 3. État réel du SEO storefront

**Documenté** — `docs/roadmap/projet-creatyss.md`, section "État actuel du socle" → "En consolidation" (ligne 47) : _"SEO storefront (balises, canonicals, sitemap — socle posé, affinement ouvert)"_.

**Observé** — éléments qui confirment un socle posé :

- `app/sitemap.ts` : génère dynamiquement un sitemap couvrant homepage (conditionnelle selon `seoSettings.sitemapIncluded`), `boutique`, `blog` (conditionnel selon flag `content.blog` niveau `publish`), les 4 pages légales statiques, les produits publiés (filtrés par `sitemapIncluded`), les articles de blog publiés (filtrés), les catégories (URLs `boutique?category=slug`). Prévoit un miroir localisé complet si le niveau `localized-routing` (L3) de la feature `platform.localization` est actif (`meetsFeatureLevel(...)`, lignes 23, 104-155).
- `app/robots.ts` : règles simples, `disallow` sur `/admin/` et `/api/admin/`, référence le sitemap.
- `app/layout.tsx` : `alternates.languages` (hreflang) générés dynamiquement, mais gatés par le niveau L3 (commentaire ligne 51 : "génère les alternates hreflang quand L3 est actif").
- `app/(public)/boutique/[slug]/page.tsx` : seule page avec `alternates.canonical` explicite (basé sur `seoCanonicalPath` produit), lignes 50-51 et 83-84.

**Observé** — éléments qui confirment un affinement encore ouvert (non clos) :

- Aucune autre page storefront observée (`boutique` listing, `blog` listing, `blog/[slug]`, `categories`, pages légales, `a-propos`, `les-marches`, `contact`) ne porte de `alternates.canonical` — seule la fiche produit en a un.
- `panier`, `checkout`, `checkout/confirmation/[reference]`, `compte` n'ont aucun export `metadata`/`generateMetadata` — ces pages héritent du titre/description génériques du layout racine (`app/layout.tsx`, `brandConfig.metadata.titleDefault`/`.description`), sans directive `noindex` explicite malgré leur nature transactionnelle/privée.
- Le hreflang (`alternates.languages`) n'a d'effet réel que si le niveau L3 `localized-routing` de `platform.localization` est actif ; ce niveau est documenté comme "cadré mais non implémenté" dans `docs/roadmap/projet-creatyss.md` Horizon 4 (ligne 215 : _"Lot 5 (`localized-routing`, L3) cadré mais non implémenté — 5 décisions produit à trancher"_) — le mécanisme existe dans le code mais n'a, à ce jour, aucun effet observable en production.
- La page `/categories` et ses URLs individuelles n'apparaissent pas dans `app/sitemap.ts` (seules les URLs `boutique?category=slug` y figurent, générées à partir des mêmes catégories mais sans lien vers `/categories` elle-même).
- Les liens de navigation storefront marqués "Bientôt" (`components/storefront/topbar/topbar-public.tsx`, lignes 314-317, item `/recherche` avec `status: "comingSoon"` ligne 99/104) confirment que la note de `projet-creatyss.md` ligne 48 (_"Liens de navigation storefront 'Bientôt' (périmètre commerce non encore ouvert)"_) est toujours d'actualité et non obsolète.

**Conclusion** : le statut documenté "en consolidation" reste **exact et non obsolète**. Le SEO storefront est **partiellement clos** : sitemap et robots sont opérationnels et couvrent le cœur catalogue/blog/légal ; les canonicals ne sont posés que sur la fiche produit ; le hreflang existe en code mais est inactif (dépendant d'un niveau de feature non livré) ; aucune page transactionnelle n'a de contrôle d'indexation explicite. Rien n'indique que ce chantier soit clos contrairement à ce que suggérerait un statut "livré" — la documentation actuelle (non modifiée dans ce lot) reflète correctement l'état observé.

## 5. Limites de cet audit

- Le formulaire cassé de `/compte` (écart #1) est établi par lecture du routing Next.js App Router, pas par exécution d'une requête HTTP réelle (environnement de dev non démarré dans ce lot) — qualifié de **déduit**, fortement probable au vu de l'absence de toute route ou middleware alternatif.
- `docs/domains/**` n'a pas été parcouru exhaustivement pour chaque route (ex. `favoris` pourrait être documentée dans une fiche de domaine non consultée dans ce lot plutôt que dans la roadmap H1-H4) — à vérifier si un lot ultérieur l'exige.
- Le contenu détaillé de `app/(public)/checkout/page.tsx` (717 lignes) n'a pas été relu ligne à ligne dans ce lot ; sa conformité s'appuie sur la référence croisée `docs/lots/2026-06-10-checkout-v1-reference.md` et le test e2e `commerce-smoke.spec.ts` déjà cités par la roadmap elle-même, pas sur une relecture complète du fichier.
- Aucune recommandation de correction n'est formulée ici, conformément au périmètre du lot. La suite probable (`lot-espace-client.md` ou équivalent) est mentionnée comme piste par le lot d'origine, non actée dans ce document.
